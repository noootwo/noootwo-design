# Noootwo Design Maintenance

Before changing this skill's aesthetics, workflow modes, stack playbooks, review gates, AGENTS integration, or claims about design quality, do a current research pass.

- Prefer official docs, framework docs, designer/developer community evidence, and artifacts with visible screenshots or running examples.
- Use X/Twitter, Reddit, HN, Figma, Flutter, React, Vue, and native app communities as signals, not as single-source truth.
- If preferred foreign sources are inaccessible, use domestic fallback sources and record access limits, source levels, and why the fallback is acceptable.
- Borrow repeatable mechanisms, not leaked prompts, proprietary wording, or surface styling.
- Borrow style craft from impeccable-style design skills as mechanism and gate guidance only; do not make target projects depend on those skills.
- Consider skill usage cost and engineering feasibility before adding gates, scripts, or required artifacts. Deep-mode requirements must not become the default path for small UI edits.
- Record the research basis in references, README notes, or implementation summaries when the change depends on external behavior.
- Do not claim a UI-quality improvement without artifact or screenshot review, and run readiness validation when `.noootwo/` deliverables are involved.
